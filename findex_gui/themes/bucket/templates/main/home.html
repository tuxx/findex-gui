About

 

Motivation

A few years ago I got the idea of writing a FTP crawler during OHM2013. Many visitors were sharing files over various protocols but unfortunately there was no central place to search for them.

It seemed like an interesting technical challenge to tackle, as writing a search engine gets you involved in many aspects of software development. First and foremost mastering Python, but also system administration and network programming.

Goals

Before I started, I established a few goals for Findex:

  • Support the following protocols: FTP, HTTP, SMB (windows), AFP (apple)
  • 100% self-hosted, no third-party assets from CDNs to guarantee privacy.
  • Should have a API that people can freely query against.
  • Searches should be fast (< 500ms), even when the database has more than 100 million rows and at the same time not impose ridiculous hardware requirements.
  • Should run on sane software: Linux, Postgres, ECMAScript 6, etc.
  • 100% open-source, MIT licensed.

Implementation

In order to reach these requirements, Findex has been designed in a modular fashion for maximum scalability. There are many components involved, some of which are listed below:

  • Postgres 9.5: The main database application.
  • RabbitMQ: AMQP message broker provides a way to manage crawl jobs. Individual crawlers subscribe on a queue and may accept or reject incoming crawl tasks.
  • ElasticSearch 1.7: While Postgres provides excellent full-text search capabilities through the GIN index, ES is better suited when it comes to Big Dataâ„¢.
  • ZomboDB: A Postgres module that syncs data to ElasticSearch.
  • Flask (Python): A microframework used by the web application. Chosen for its simplicity.
  • Twisted (Python): Asynchronous networking library that powers the crawl bots.

What's next?

Improved metadata capabilities could possibly catapult its suitability for more enterprise use-cases. For now ill settle with trying to index as many servers as possible.
